# リポジトリ初期ドキュメント整備

## 概要

このスキルは、まだ本格的な構成が整っていない新規リポジトリに、最小限で運用しやすい文書基盤を作るためのものです。対象は、`README.md`、`.gitignore`、簡単なプレースホルダーだけが置かれた初期状態のリポジトリを想定します。

目標は、次のような小さくて持続しやすい初期構成を整えることです。

- `README.md` と `README-ja.md`
- `AGENTS.md` と `AGENTS-ja.md`
- 日英の対応文書を同じ変更の中でそろえて更新する、というルール

既存ファイルがある場合は、置き換えるより活かして広げることを優先します。ユーザーから大きな再構成を求められていない限り、いまある文書や命名、構成を前提に最小限の整備を行います。

## 使う場面

次のような依頼で使います。

- 新しいリポジトリの文書を最初に整えたい
- 最小構成のリポジトリに `AGENTS.md` を追加したい
- README や AGENTS に英語版と日本語版の対応ファイルを用意したい
- 実装が増える前に、軽量な文書運用ルールを先に置きたい
- まばらな初期状態のリポジトリを、今後エージェントが扱いやすい基準線まで整えたい

すでに文書体系が十分に整っているリポジトリには、ユーザーが明示的に整理し直したいと言っていない限り使いません。

## 前提

- リポジトリは、まだ最低限のファイルしかない可能性がある。
- 既存ファイルは誤っているというより、未完成であることが多い。
- ユーザーが求めているのは、過不足のない初期整備であり、大規模な文書一式ではない。
- 日英同期の方針は、後から付け足すより早い段階で定着させたほうがよい。

## 最初に確認するファイル

編集前に、まずリポジトリ直下で次を確認します。

- `README.md`
- `README-ja.md`
- `AGENTS.md`
- `AGENTS-ja.md`
- `.gitignore`
- `package.json`、`pyproject.toml`、`go.mod`、`Cargo.toml`、`mise.toml`、`apm.yml` など、用途や技術スタックの手がかりになる明白なマニフェスト
- リポジトリの目的が見えるトップレベルディレクトリ

既存内容から、少なくとも次を判断します。

- このリポジトリが何のためのものか
- 技術スタックやツールチェーンがすでに明確か
- 日英の対応ファイルがすでに存在するか
- 既存文言を活かして追記すべきか、単なる置き換えでは不自然か

## ワークフロー

1. リポジトリの状態を確認し、基準となる文書ファイルの有無を把握する。
2. 現在のファイルやディレクトリ名から、リポジトリの目的を推測する。誤った推測が文書をミスリードしそうなら、先にユーザーへ確認する。
3. 使える既存文言は残す。短いプレースホルダーは全面的に捨てるのではなく、必要な情報を補って育てる。
4. 主要文書について、存在するなら必ず日英の対応をそろえる。
   - `README.md` と `README-ja.md`
   - `AGENTS.md` と `AGENTS-ja.md`
5. `AGENTS.md` には、英語版と日本語版の対応文書を同じ変更で更新し、意味の対応を保つことを明記する。
6. 日本語ファイルは、日本語として自然に書く。意図はそろえるが、逐語訳にはしない。
7. 初回の整備は意図的に小さく保つ。推測に基づいて文書群を膨らませるより、明確な出発点を作ることを優先する。
8. 最後に、作成または更新したファイルと、初期文書の内容を決める際に置いた前提を報告する。

## `README` に入れる内容

最小構成のリポジトリに対する `README.md` と `README-ja.md` は、通常は次の要素だけで十分です。

- このリポジトリの目的
- まだ初期化中であれば、その状態
- 必要ならトップレベル構成の簡単な説明
- 自信を持って書ける範囲での作業開始方法
- リポジトリ固有のエージェント向け指示がどこにあるかの短い案内

リポジトリの内容やユーザーの依頼から裏付けられないセットアップ手順、設計方針、リリース運用は書き足しません。

## `AGENTS` に入れる内容

最小構成のリポジトリに対する `AGENTS.md` と `AGENTS-ja.md` には、今後も残りやすい基礎ルールだけを入れます。典型的には次のような内容です。

- ユーザー依頼、プロジェクト文書、グローバル既定の優先順位
- 日英の対応文書を同期して保つルール
- 変更は小さく、依頼範囲に収めるという原則
- 破壊的操作、秘密情報、Git 操作に関する安全上の扱い
- まとまった変更の前に、既存ファイルを読んで文脈を確認すること

`AGENTS.md` は、リポジトリの初期成長に耐えられる程度に汎用的に保ちます。リポジトリから明確に読み取れない限り、技術スタック依存の細則は埋め込みません。

## 編集方針

- まずは最小限で完結した文書基盤を作る。
- 既存の名称、用語、ディレクトリ構成を尊重する。
- 現時点で必要性がない補助ファイルは増やさない。
- 他のリポジトリでよく見るという理由だけで、大きな方針節を追加しない。
- すでにより強い文書構成がある場合は、この初期形に無理に寄せず、既存構成に合わせて整える。

## 非ゴール

このスキルだけでは、次のことまでは行いません。

- 完全な contributing guide、ADR 群、オンボーディング文書の設計
- issue template、PR template、CI workflow、license ファイルの作成
- 根拠のない技術スタックの決め打ち
- 成熟した文書体系の無断での再編成

## 最終回答

結果を報告するときは、次を含めます。

- どの基準ファイルを作成または更新したか
- 日英の対応ファイルを追加したか、同期したか
- リポジトリの目的や成熟度について置いた前提
- 実行していない確認や検証があれば、その旨
