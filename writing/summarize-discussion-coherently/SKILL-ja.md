# 議論を一貫した形で要約する

## 概要

個人の対話記録、往復の多い議論、断片的なメモを、読み手が理解しやすい Markdown 文書に組み直す。単に短くするのではなく、その対話で何を前に進めようとしていたのか、どこまで整理できたのか、何が根拠になっていたのか、何がまだ残っているのかが伝わる形に再構成する。

## ワークフロー

1. 会話を必要な範囲で読み、対話の主目的、現時点の合意、説明中心か、検討中心か、意思決定中心かを把握する。
2. 書き始める前に、もっとも理解しやすく整理できるフレームを推定する。形式張って見えるものではなく、内容に最も合うものを選ぶ。
3. 重要な情報が欠けていて、解釈によって要約の意味が大きく変わる場合だけ、簡潔な確認質問を最大 3 問まで行う。そうでなければ、不確かな点を明示したうえで先に進む。
4. 内容は時系列のまま並べるのではなく、論点、根拠、帰結の単位で再構成する。時系列が重要な場合だけ、その順序を残す。
5. 結論だけでなく、そこに至る目的、前提、制約、トレードオフ、未整理点も必要に応じて残す。
6. 結果は、そのまま `.md` ファイルとして保存できる Markdown 文書として出力する。
7. 出力の最後に、要約が一貫しているか、判断や次の作業に使えるか、元の対話を取り違えていないかを短くセルフチェックする。

## フレームの選び方

主となる整理フレームを 1 つ選び、出力の中で明示する。

| フレーム | 向いている場面 | 強み |
| --- | --- | --- |
| `PREP` | ひとつの主張を明確に着地させ、その理由と補足例を添えて短く説得的にまとめたいとき。 | 短くても筋の通った説明にしやすい。 |
| `TAPS` | 話題や緊張点から分析、提案、次の一手へ進めたいとき。 | 議論を行動につながる流れにしやすい。 |
| `SCQA` | 背景、複雑化した事情、中心となる問い、答えの順で流れを作りたいとき。 | なぜこの話が重要なのかを示したうえで結論に入れる。 |
| `時系列の整理` | 後の発言で前提が変わるなど、発話順そのものが意味を持つとき。 | 順序を保ちながら、読みやすく整理できる。 |
| `論点整理メモ` | 複数の懸念、前提、反論、未解決の問いが混ざっていて、まず切り分けが必要なとき。 | もつれた議論を論点ごとに分離しやすい。 |
| `意思決定メモ` | 選択肢、判断基準、トレードオフ、推奨案を整理したいとき。 | 推奨とその根拠を残しやすい。 |

複数の型を補助的に使ってもよいが、最終的な文書には主となる整理軸を 1 つ持たせる。

## 出力契約

すべての対話に同じ固定テンプレートを当てはめない。見出し名や並び順は選んだフレームに応じて変えてよい。ただし、次の要素は文書内のどこかに必ず含める。

- 採用した整理フレーム
- 現時点の結論、または最善の要約
- その対話で何を整理しようとしていたのかという目的
- 議論の方向を決めた背景、制約、前提
- 検討された主な論点、主張、選択肢
- 未整理の点、不足情報、判断保留の点
- 会話から自然に導かれる次の一手

## スタイル

- 何でも詰め込むのではなく、理解しやすさを優先する。
- 時系列よりも構造を優先する。
- 確定したことと、解釈、推定、未解決の点を分けて書く。
- 元の会話が散らかっている場合は、その散らかり方がわかるように整理し、無理にきれいにしすぎない。
- 見出しは直接的にし、本文は簡潔に書く。
- 出力は Markdown に保ち、装飾的な書き方は避ける。

## 最終回答

ユーザーから別の指定がない限り、回答本文はユーザーの言語に合わせる。出力は Markdown 文書そのものを返す。重要な不確実性が残る場合は、補足の会話でぼかさず、文書中に `未確認` や `要確認` として明示する。

## 出力例の形

### 例: PREP

```md
# なぜラッパースキルを作るべきか

## 採用フレーム
PREP

## Point
既存の議事録系スキルを土台にして、ラッパーを作るのが最適である。

## Reason
既存スキルは Markdown 要約までは強いが、個人の対話を目的、根拠、未整理点の単位で再構成するところは十分ではない。

## Example
今回の議論では、結論だけでなく背景や論点整理まで求められていたため、通常の meeting minutes では不足した。

## Point の言い換え
要約だけでなく整理フレーム選択まで含むラッパーを作るべきである。

## 未整理の点
- どのフレーム名をユーザー向けに公開するか
```

### 例: 意思決定メモ

```md
# 新しいスキルの議論まとめ

## 採用フレーム
意思決定メモ

## 現時点の結論
スキル名は `summarize-discussion-coherently` とし、出力は最小限の契約だけ固定し、本文構造はフレームごとに変える。

## 目的
個人の対話ログから、一貫した Markdown の要約文書を作る再利用可能なスキルを定義する。

## 検討した案
### 固定テンプレート
標準化しやすいが、フレームを選ぶ意味が薄れる。

### 最小契約 + フレーム別構成
一貫性を保ちながら、選んだ整理フレームを活かせる。

## 推奨
最小契約方式を採用する。

## 未整理の点
- 既定の見出し構成をどこまで強く誘導するか

## 次の一手
- `SKILL.md` と `SKILL-ja.md` を作成する
```
