# Cleanup English Writing

## 概要

技術ドキュメント、コメント、プロンプト、ポリシー、リリースノート、ガイドの英語表現を整える。
`stop-slop` を、AI っぽさを取り除くための基礎チェックとして扱い、そのうえでこのスキルの英語技術文書向けの確認観点を適用する。
このスキルと `stop-slop` が矛盾する場合は、`stop-slop` を優先する。

目標は、明確かつ具体的で人間が書いたように読める英語にすること。
実用的なドキュメントをマーケティング文体に寄せたり、文体のために技術的な意味を変えたりしない。

## 手順

1. ユーザーの依頼から対象テキストを特定する。範囲が指定されていない場合は、関連する差分やファイルを確認する。
2. 普通の文章と、文字どおり保持すべき技術要素を分ける。
   - 普通の文章: 文、見出し、箇条書き、説明、ラベル、コメント、プロンプト、解説。
   - 技術要素: コマンド、ファイルパス、設定キー、コード識別子、パッケージ名、API 名、URL、例、引用値、製品名。
3. `stop-slop` が使える場合は、AI っぽさを取り除くための基礎チェックとして適用する。そのチェックリストをこのスキル内で重複させない。
4. このスキルの技術文書向けの確認観点を適用する。
5. ユーザーが明示的に変更を求めていない限り、技術要素は維持する。
6. リポジトリ内の既存の用語、大文字小文字、文体に合わせる。
7. ユーザーが大きな編集を求めていない限り、文書の構成や意図を変えるような広い書き換えは避ける。
8. 編集後、対象にした不自然な語や繰り返し表現を検索し、意図した整理が完了していることを確認する。

## 英語技術文書の指針

直接的で具体的な英語を優先する。

| 避ける表現                         | 推奨する表現                                           |
| ---------------------------------- | ------------------------------------------------------ |
| `utilize`                          | `use`                                                  |
| `leverage`                         | 文脈に応じて `use`、`build on`、またはより具体的な動詞 |
| `ensure that`                      | `ensure`、または直接的な指示                           |
| `in order to`                      | `to`                                                   |
| `it is important to note that`     | その後に続く要点そのもの                               |
| `seamlessly`, `robustly`, `easily` | 具体的な動作や制約                                     |
| `delve into`, `dive into`          | `inspect`、`read`、`analyze`、`review`                 |
| `not only X but also Y`            | 型どおりの構文を外した具体的な要点                     |

技術文書として地に足のついた表現にする。

- 判断、作業、責任を述べる文では、できるだけ行為者を明確にする。
- 抽象名詞より動詞を優先する。`runner configuration is performed` ではなく `configure the runner` と書く。
- 曖昧な主張は、具体的な動作、リスク、トレードオフに置き換える。
- 見出しや箇条書きは簡潔にする。ただし意味が読み取れないほど短くしない。
- 一般的な表記ルールと違っていても、プロジェクトで定着している用語を優先する。
- 例、コマンド、フラグ、パス、識別子、引用値は正確に残す。
- contractions は、周囲の文書の文体に合う場合だけ使う。
- 元の文章にない熱量、売り文句、広すぎる主張を足さない。

## プロジェクト固有の確認観点

このスキルを育てるときは、`stop-slop` を直接変更したり言い換えて再掲したりせず、安定した確認観点をここに追加する。

- 箇条書き、例、結論の前にある前置きだけの文を削る。
- `improves developer experience` のような一般的な主張は、具体的に何が改善されるのかに置き換える。
- 変更がより賢い、安全、洗練されているといった表現は、本文で根拠を示せる場合だけ使う。
- コミットメッセージ、PR 説明、changelog は簡潔にし、結果に焦点を当てる。
- プロンプトやエージェント指示では、義務を検証可能にする。何を、いつ、どう確認するかを書く。

## 確認観点

- 慎重なメンテナが書いた文章として読めるか
- 文章を硬くしすぎずに、ありがちな AI っぽい表現を削れているか
- 主張は検証できる程度に具体的か
- 文体変更によって技術的な意味が変わっていないか
- コマンド、パス、識別子、設定キー、例、製品名が正確に残っているか
- 周辺ファイルと用語、大文字小文字、文体が揃っているか
- マーケティング調や根拠のない主張を足していないか
- ファイルを対象にした依頼では、編集後に対象の整理が完了していることを確認したか

## 出力

ファイルを編集した場合は、最後にどの種類の表現を変更したかを簡潔にまとめる。編集が不要な場合は、文章がすでに十分明確であることと、意図的に維持した表現を伝える。
