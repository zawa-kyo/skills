# 本質的なコードレビュー

## 言語バージョン

このファイルと `SKILL.md` は同じ skill を説明するものです。どちらかを変更する場合は、同じ変更の中でもう一方も更新し、英語版と日本語版の内容をそろえます。その際、それぞれの言語として自然な表現を保ちます。

## 概要

まず通常のコードレビューの観点で、正しさ、リグレッション、エッジケース、セキュリティ、パフォーマンス、テストカバレッジを確認します。そのうえで、さらに重要な観点として単純さを加えます。つまり、そのコードがローカルな設計の流儀に沿って根本問題を解いているのか、それとも特殊ケース、重複ロジック、広すぎるガード、不要な抽象化などで症状だけを塞いでいるのかを確認します。

## ワークフロー

1. ユーザーの依頼からレビュー対象を特定する。対象が指定されていない場合は、ユーザーが staged-only やより狭い対象を指定していない限り、現在の Git 差分をステージング済みと未ステージングに分けて確認する。
2. 既存の設計、不変条件、公開契約、ローカルスタイルを理解するために、十分な周辺コードを読む。文脈なしに単純さを判断しない。
3. ユーザーが明示的に修正を依頼していない限り、レビュー中にワークツリーや index を変更しない。特に、`git add`、`git restore --staged`、フォーマット、その他レビュー対象の差分を stage、unstage、書き換え、正規化するコマンドを実行しない。
4. まずリスクレビューを実施する。バグ、リグレッション、欠けているエッジケース、壊れた API 契約、データ損失、セキュリティ問題、パフォーマンス問題、不足、脆弱なテスト、スタイルなどを確認する。
5. 次に、本質的な設計をレビューする。
   - 変更は根本原因に対処しているか、あるいは観測された失敗を抑え込んでいるだけか。
   - 既存の不変条件を保っているか、あるいは散らばった例外処理を増やしているか。
   - 偶発的な複雑さを取り除いているか、あるいはテストを通すために無理をして flag、分岐、retry、catch、sleep、cast、global、重複ロジックを増やしているだけか。
   - その抽象化は、実際の再利用や複雑さの低減によって正当化されているか。
   - 呼び出し側で挙動が明確で、実装者以外の開発者やメンテナーが理解しやすいか。
6. 具体的なコード箇所と問題たる理由を指摘することを優先する。保守性に影響するか、本当のリスクを隠している場合を除き、スタイル上の細かな指摘は避ける。
7. 指摘が暗黙の契約に依存する場合は、findings の後でその前提を明示する。その前提が弱く、リスクが仮説に留まる場合は、finding として断定せず open question に落とす。
8. 実装が単純で堅実な場合は、そう明確に伝える。レビュー形式を満たすために指摘を作り出さない。

## 重大度と信頼度

- 重大度は、コードが怪しく見える度合いではなく、ユーザーに見える影響と発生しやすさで調整する。データ損失、セキュリティ露出、公開契約の破壊、通常利用でのクラッシュは多くの場合、重要度が高いものである。一方、テスト不足や保守性の懸念は、具体的なリグレッションを可能にしている場合を除き、それより低優先度として扱う。
- レビュー対象が概念上の diff、抜粋など不完全な材料である場合は、信頼度に関わる未検証の点を明記する。文脈が不完全でも、どのように問題が発生するかを説明できる指摘は控えてはならない。ただし、その指摘が依存している前提は明示する。
- 修正案はまず最小限のものを提示する。大きな修正を提案するより、既存の不変条件やその挙動を所有する境界を復元する方針を優先する。しかし本質的に問題が残る場合は、大きな修正案を提案することも遠慮せず行う。

## 回避策の匂い

以下は自動的な欠陥ではなく、より深く確認するための手がかりとして扱う。

- 広すぎる `try/catch`、`return null`、デフォルト値、または guard によって、なぜ続行してよいのかを説明しないまま不正な状態を隠している。
- 近くのコードに既にある一般ルールを使わず、失敗した入力だけをハードコードで特別扱いしている。
- 変換や検証のロジックを、その責務を持つドメイン境界で再利用せずに重複させている。
- ある失敗経路を迂回するためだけに、新しい設定、flag、任意パラメータを追加している。
- 実証された並行性モデルやライフサイクルモデルなしに、タイミング依存の修正、retry、sleep、cache clear、順序依存を追加している。
- type assertion、未検査の cast、緩めた schema によって、契約を弱めながらコンパイラや validator の警告を止めている。
- テスト変更が assertion を減らす、挙動を skip する、または意図した挙動を証明せずに新しい出力を正解として固定している。

## 出力

標準的なコードレビューの形を使います。

- 重大度順に findings から始める。
- 各 finding には、ファイルと行の参照、重大度、具体的なリスク、最小限の方向性として正しい修正方針を含める。
- 読みやすくなる場合に限り、「correctness findings」と「simplicity/design findings」を分ける。
- findings の後に、未解決の質問や前提を書く。
- レビューの信頼度に影響する場合は、実行しなかった検証を明記する。
- 簡潔な summary は findings の後にだけ置く。
- ユーザーが明示的に別言語を求めていない限り、レビューはユーザーの依頼言語で書く。

問題が見つからない場合は、blocking issue は見つからなかったと明確に伝え、残るテストや文脈上の不足があれば触れます。

ユーザーが明示的に修正を依頼していない限り、コードを書き換えることはしない。レビュー結果の出力は、ユーザーがその理由を理解する助けとなるため、背景や理由を十分に、かつ明瞭に説明する。
