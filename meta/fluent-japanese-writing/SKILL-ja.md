# Fluent Japanese Writing

## 言語版

このファイルと `SKILL.md` は同じスキルを説明するものです。どちらかを変更する場合は、同じ変更の中でもう一方も更新し、英語版と日本語版の内容をそろえます。その際、それぞれの言語として自然な表現を保ちます。

## 概要

技術ドキュメント、コメント、ポリシー、ガイドの日本語表現を整える。すべての技術語を機械的に翻訳するのではなく、技術的な意味を保ったまま、日本語話者のメンテナが読みやすい文章にする。

## 手順

1. ユーザーの依頼から対象テキストを特定する。範囲が指定されていない場合は、関連する差分やファイルを確認する。
2. 普通の文章と、文字どおり保持すべき技術要素を分ける。
   - 普通の文章: 文、見出し、箇条書き、説明、ラベル。
   - 技術要素: コマンド名、ファイルパス、設定キー、コード識別子、パッケージ名、URL、例、引用値。
3. 普通の文章を読みやすい日本語に書き換える。
4. 技術要素は、ユーザーが明示的に変更を求めていない限り維持する。
5. リポジトリ内の既存の用語と文体に合わせる。
6. ユーザーが大きな編集を求めていない限り、文書の構成や意図を変えるような広い書き換えは避ける。
7. 編集後、対象にした不自然な語を検索し、意図した整理が完了していることを確認する。

## 表記の指針

普通の文章では読みやすい日本語を優先する。

| 元の語               | 推奨する日本語                                              |
| -------------------- | ----------------------------------------------------------- |
| `repository`         | `リポジトリ`                                                |
| `public repository`  | 文脈に応じて `パブリックリポジトリ` または `公開リポジトリ` |
| `private repository` | `プライベートリポジトリ`                                    |
| `skill`              | `スキル`                                                    |
| `task`               | `タスク`                                                    |
| `shell function`     | `シェル関数`                                                |
| `shell script`       | `シェルスクリプト`                                          |
| `formatter`          | `フォーマッタ`                                              |
| `editor`             | `エディタ`                                                  |
| `token`              | `トークン`                                                  |
| `user scope`         | `ユーザー単位`                                              |
| `project scope`      | `プロジェクト単位`                                          |
| `dependency`         | `依存関係`                                                  |
| `package`            | `パッケージ`                                                |
| `tag`                | `タグ`                                                      |
| `commit`             | `コミット`                                                  |

リポジトリ固有の好みがある場合はそれを優先する。

- ローカルの表記として定着している場合は、`ロックファイル` ではなく `lock ファイル` を使う。
- `正本` のように硬すぎたり AI っぽく見えたりする語は、より平易な語で足りるなら避ける。
- 文脈に応じて `管理元`、`置き場所`、`参照先`、`基準`、`まとめるディレクトリ`、`まとめるファイル` などを使う。
- `ポリシー文書` のように意味が重複する表現は、`ポリシー` で足りるなら短くする。

文字どおり保持すべき技術要素は変更しない。

- コマンド: `mise run install`, `uv run pre-commit run -a`
- ファイルパス: `docs/index.md`, `.apm/skills`
- 設定キー: `dependencies`, `interface.default_prompt`
- スキル名やパッケージ名: `review-essential-code`, `suggest-commit-messages`
- 製品名や固有名詞: GitHub, Homebrew, Claude Code, Codex

## 確認観点

- 慎重な日本語話者のメンテナが書いたドキュメントとして読みやすいか
- 不要な英語が読みやすい日本語に置き換わっているか
- コマンド、パス、識別子、設定キー、例が正確に残っているか
- 表現変更によって技術的な意味が変わっていないか
- 編集したファイル間で用語が揃っているか
- 固有名詞やコード風の文字列まで誤って置き換えていないか

## 出力

ファイルを編集した場合は、最後にどの種類の表記を変更したかを簡潔にまとめる。編集が不要な場合は、文章がすでに十分読みやすいことと、意図的に維持した用語を伝える。
