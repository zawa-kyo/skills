# Edit Existing Document

## 目的

既存の文章を、依頼された箇所だけのパッチとして扱わずに編集する。文書やコメントを変える前に、文書の目的、読者、セクション構成、周辺説明を確認し、全体の流れに合う形で変更する。

このスキルは、構成判断と既存文書への統合方針を担当する。言語別の文体スキルの代わりにはしない。日本語の本文では `revise-japanese-writing` を使い、英語の本文では使える場合に `cleanup-english-writing` や `stop-slop` を併用する。

## 対象

次のような既存テキストを扱うときに使う。

- `README.md`、ガイド、ポリシー、設計文書、運用手順。
- `AGENTS.md`、`CLAUDE.md`、プロンプト、エージェント向け指示。
- コードコメントや、設定ファイル内の説明文。
- 既存文書への追記、修正、移動、統合、構成整理を伴うタスク。

新規文書を作る場合でも、既存の文書体系に組み込む必要があるならこのスキルを使う。

## 対象外

次の判断の基準としては使わない。

- 日本語文体の細かな判断。`revise-japanese-writing` を使う。
- 英語文体の細かな判断。`cleanup-english-writing` や `stop-slop` を使う。
- `.docx`、プレゼンテーション、スプレッドシートのレンダリング確認。該当する文書、プレゼンテーション、スプレッドシート用のスキルを使う。
- コード品質レビュー。`review-essential-code` を使う。
- プロジェクト固有の配置ルール。最も近い `AGENTS.md`、リポジトリ文書、既存の慣習に従う。

## 手順

1. 依頼された変更内容と対象ファイルを特定する。対象を安全に推測できない場合は、編集前に確認する。
2. 対象セクションと、流れを理解するために必要な周辺説明を読む。
3. 変更先を決める前に、文書全体の見出し構成を確認する。
4. 近いセクションに同じ内容、矛盾する内容、粒度の違う説明がないかを確認する。
5. 小さな追記、既存文の修正、移動、統合、より大きな構成整理のどれが適切かを判断する。
6. 英日版の対応文書、生成メタデータ、関連する例など、同期すべきファイルがないか確認する。
7. 大きな構成整理が不要な場合は、文書の流れを保ち、重複を増やさない最小の編集を行う。
8. 自然文を編集した場合は、使える範囲で対象言語の文体スキルを併用する。
9. リポジトリにフォーマッタや検証コマンドが用意されていれば実行する。実行しない場合は理由を報告する。

## 構成整理の確認

次のいずれかに当てはまる場合は、ユーザーに確認せずに構成整理を始めない。

- 依頼された内容を入れると、対象セクションの責務が広がりすぎる。
- 近い内容が複数のセクションに分散しており、追記すると重複が増える。
- 新しい内容が現在の見出し構成に自然に収まらない。
- 既存セクションを移動または統合したほうが文書として読みやすい。
- 複数ファイルにまたがる構成変更が必要になる。
- 最適な編集がユーザーの依頼範囲を超えそうである。

この条件に当てはまる場合は、先に再整理案を簡潔に示す。理由、影響するファイル、想定する見出しや移動先、ユーザーに確認したい点を含める。確認後は、合意した構成に範囲を絞って編集する。

## 編集方針

- セクション末尾に新しい段落を足すよりも、既存の説明に自然に統合することを優先する。
- 文書の目的と読者を保つ。高レベルなポリシーに、既存の粒度と合わない運用手順を足さない。
- すでに近くにある説明を繰り返さない。必要なら既存の説明を修正するか、そこを参照する。
- 用語、見出しの立て方、箇条書きの形、例、相互参照を周辺文書に合わせる。
- 後から順序や内容が変わりそうな箇所では、見出しや本文に固定の番号を埋め込まない。順序が必要な場合は Markdown のリストを使う。
- 依頼された編集を一貫させるために必要な場合を除き、無関係なセクションは触らない。

## 出力

結果を報告するときは、変更したファイル、対応する言語版や関連ファイルを同期したか、実行したフォーマッタや検証コマンドを伝える。編集せずに構成整理を提案した場合は、提案した構成をまとめ、ユーザーの判断を待つ。
